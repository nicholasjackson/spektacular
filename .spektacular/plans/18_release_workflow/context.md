# Context: 18_release_workflow

## Current State Analysis

**Existing Build Infrastructure**:
- `Makefile:19-23` — Cross-compilation targets for darwin/linux on amd64/arm64 using `CGO_ENABLED=0`
- `.github/workflows/build.yml:1-38` — Current workflow runs `make lint`, `make test`, and `make cross` on every push
- `go.mod:3` — Go version 1.25.0
- No `dagger/` directory exists in the repository yet
- No release automation exists — users must clone and build manually

**Jumppad's Dagger Module** (reference implementation):
- `jumppad-labs/jumppad/dagger/main.go` — exported `JumppadCI` type with `All`, `Release`, `Build`, `UnitTest`, `Package`, `Archive`, `GenerateChecksums`, `SignAndNotorize`, `GithubRelease`, `UpdateBrew`, `UpdateGemFury`, `UpdateWebsite`, and functional test helpers
- Uses error chaining pattern with `lastError` and a shared Go cache volume
- Creates Linux `.deb` packages via Dagger `cli.Deb()`; does not use rpm/fpm/nfpm
- Archives darwin artifacts, uploads release assets, updates Homebrew, pushes GemFury packages, and updates a website version file
- Spektacular must port the implementation shape but remove website, Windows, winget, and functional-test workflow concerns per spec

**Secret Requirements**:
- Seven GitHub org-level secrets must exist: `GH_TOKEN`, `FURY_TOKEN`, `QUILL_SIGN_P12`, `QUILL_SIGN_PASSWORD`, `QUILL_NOTORY_KEY`, `QUILL_NOTARY_KEY_ID`, `QUILL_NOTARY_ISSUER`
- These names are fixed by the spec and must be consumed as-is so org-level secrets can be reused

- Exclude functional-test, podman, discord, sync_winget, and winget jobs

Representative shape:
```yaml

on:
  push:
    branches:
      - '**'

  dagger_build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: dagger/dagger-for-github@v8.2.0
        with:
          version: "0.19.8"
      - name: Build
        run: |
          dagger call all ... export --path=./archives
      - uses: actions/upload-artifact@v4
        with:
          name: archives
          path: archives/
```

### Phase 4.2: Release Workflow

Add the `release` job to `.github/workflows/build_and_deploy.yaml`.

Implementation notes:
- Gate with `if: ${{ github.ref == 'refs/heads/main' }}`
- Download the `archives` artifact
- Invoke the Dagger `Release` function with `GH_TOKEN` and `FURY_TOKEN`
- Write the returned version to `./version.txt`
- Expose the version via `outputs.version`
- Non-main branch pushes should show the job as skipped

Representative shape:
```yaml
release:
  needs: dagger_build
  if: ${{ github.ref == 'refs/heads/main' }}
  outputs:
    version: ${{ steps.release.outputs.version }}
  steps:
    - uses: actions/checkout@v4
    - uses: actions/download-artifact@v4
      with:
        name: archives
        path: archives/
    - uses: dagger/dagger-for-github@v8.2.0
      with:
        version: "0.19.8"
    - id: release
      run: |
        VERSION=$(dagger call release ...)
        printf "%s" "$VERSION" > ./version.txt
        echo "version=$VERSION" >> "$GITHUB_OUTPUT"
```

**Complexity**: Low  
**Token estimate**: ~5k  
**Agent strategy**: Single agent, sequential execution

## Testing Strategy

Testing is primarily workflow- and artifact-oriented:
1. Run `dagger functions` against `./dagger` and verify `all` and `release` are present
2. Run `dagger call all` locally with/without notarization inputs to verify success/failure behavior
3. Extract output tarballs and verify each contains exactly one binary named `spektacular`
4. Verify Linux `.deb` packages are produced for GemFury publishing
5. After a main-branch release, verify GitHub Release assets, Homebrew formula update, and GemFury package presence
6. Confirm `.github/workflows/build_and_deploy.yaml` contains no excluded jobs and that non-main pushes skip `release`
